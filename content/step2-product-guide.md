# Phase 0: Product팀 조직 소개

## EXPLAIN

### 제품 체험 전에, 이 제품을 만드는 사람들을 먼저 알아볼까요?

콜라보를 만들고 운영하는 조직은 **Product팀**입니다.
CPO **Alex**가 이끌고 있으며, 아래 직군으로 구성되어 있어요.

| 직군 | 하는 일 |
|------|---------|
| **PM** (프로덕트 매니저) | 제품 기획, 스쿼드 리딩, 로드맵 관리 |
| **PD** (프로덕트 디자이너) | UI/UX 디자인, 프로토타이핑, 디자인 시스템 |
| **QA** (품질 보증) | 테스트 자동화, 품질 관리, 장애 대응 |

### 스쿼드 체제

Product팀에서 실제 프로덕트를 만들 때는 **스쿼드(프로젝트 단위 팀)** 체제로 일합니다.
하나의 스쿼드에 PM + PD + 개발(BE/FE) + QA가 함께 모여서, 하나의 미션을 수행해요.

현재 운영 중인 스쿼드:
- **Realtime 스쿼드**: 화이트보드 실시간 동시 편집 성능 고도화
- **Board 스쿼드**: 프로젝트 보드 기능 확장 (간트 차트, 의존성 관리)
- **Growth 스쿼드**: 온보딩 퍼널 최적화, 무료→Pro 전환율 개선
- **AI 스쿼드**: AI Copilot "Cody" 개발 (회의록 자동 생성, 태스크 추천)

> ⚠️ Engineering팀(CTO: Leo)은 별도 조직이며, **일부 엔지니어가 스쿼드에 소속**되어 Product팀과 함께 일합니다.
> 모든 엔지니어가 스쿼드에 있는 건 아니에요 — **테크 이니셔티브**(기술 주도 프로젝트)를 중점으로 하는 경우도 있습니다.
> 예: Platform셀 리드 **Sam**, Frontend셀 리드 **Noah**, Backend셀 리드 **Kai**는 스쿼드에 소속되어 있지 않아요.

# Phase 1: 워크스페이스 오너 + 멤버 체험

## EXPLAIN

### 개밥먹기(Dogfooding)란?

"개밥먹기"는 자기가 만든 제품을 직접 써보는 것입니다.
우리 제품을 직접 써봐야 고객에게 제대로 설명할 수 있고, 진짜 개선점을 발견할 수 있어요.

콜라보에는 두 종류의 핵심 사용자가 있습니다:
- **워크스페이스 오너(Admin/Owner)**: 팀 워크스페이스를 만들고 멤버·권한·보드 구조를 설계하는 관리자
- **협업 멤버(Member)**: 화이트보드와 프로젝트 보드에서 아이디어를 만들고 태스크를 실행하는 팀 구성원

직무와 관계없이, 모든 구성원이 **양쪽 관점**을 이해해야 합니다.

### 체험 + 기록을 동시에

이번 Phase에서는 체험과 기록을 **동시에** 진행합니다.
Notion에 **개밥먹기 노션 문서**를 열어두고, 체험하면서 느낀 점을 바로바로 적어주세요.

나중에 한꺼번에 쓰면 기억이 흐려지지만, **체험하면서 바로 적으면** 생생한 인사이트가 남아요!

기록할 때 아래 관점을 참고하세요:
- **막힌 부분**: 어디서 헤맸는지, 뭐가 헷갈렸는지
- **원활한 부분**: 어디가 직관적이고 쉬웠는지
- **설계 의도 추정**: "왜 이렇게 만들었을까?" 나름의 추측
- **워크스페이스 오너 vs 협업 멤버 비교**: 양쪽 경험의 차이점

---

## EXECUTE

> 이번 Phase에서는 🌐 **Notion 브라우저**(체험 가이드 + 개밥먹기 기록)와 🌐 **콜라보/Collabo Academy 사이트**(체험)를 사용합니다. 체험이 끝나면 Claude Code로 돌아와주세요.

> **에이전트 지시** (신규입사자에게 보여주지 않는 내부 메모):
>
> **⚠️ 체험 가이드와 개밥먹기 문서는 서로 다른 페이지다. 반드시 구분하여 안내한다.**
>
> **1단계: 체험 가이드 열기** (공용 문서 — 모든 대상자가 같은 페이지)
> - `notion-fetch`로 Step 2 Product Onboarding 페이지(`step2_product` ID)를 fetch
> - 페이지 URL을 브라우저로 열어준다
> - 직접 가이드 내용을 요약하지 않는다 — **Notion 페이지 링크로 안내**하여 원본을 직접 읽게 한다
>
> **2단계: 개밥먹기 문서 열기** (개인 문서 — 대상자별 다른 페이지)
> - `progress.json`의 `dogfooding_page_id`를 확인한다
> - ID가 있으면: 해당 페이지 URL을 브라우저로 열어준다
> - ID가 null이면: `notion-search`로 "{닉네임} 개밥먹기"로 재검색 → 찾으면 progress.json 업데이트
> - 그래도 못 찾으면: "HR(Dana)에게 개밥먹기 노션 문서를 요청해주세요" 안내
>
> **금지 사항:**
> - fetch 실패 시 fallback: "Notion에서 'Product Onboarding'을 검색하면 체험 가이드를 찾을 수 있어요"
> - ⚠️ 콜라보를 강의 마켓플레이스/노코드 사이트 빌더처럼 설명하지 않는다. "강의 신청", "수강", "사이트 만들기" 식의 안내 금지.
>   콜라보는 팀이 화이트보드에서 만든 아이디어를 프로젝트 보드의 실행 태스크로 연결하는 실시간 협업 SaaS다.

### 체험 순서

**두 개 탭**을 Notion에서 열어두세요:
- **체험 가이드** — 체험 순서를 따라가는 용도
- **개밥먹기 노션 문서** — 느낀 점을 바로 적는 용도

**미션 1: 워크스페이스 오너 체험**
1. 회사 계정이 아닌 **개인 계정**으로 콜라보에 회원가입
2. 워크스페이스 오너로서 워크스페이스를 만들고 멤버 초대·권한·보드 구조를 설정해보세요 (Notion 체험 가이드 참고)
3. 체험하면서 막힌 점/원활한 점을 **개밥먹기 노션 문서에 바로 기록**

**미션 2: 협업 멤버 체험**
1. 샘플 워크스페이스에서 멤버 관점으로 보드에 참여해보세요
2. 화이트보드 아이디어 작성 → 태스크 전환 → 프로젝트 보드 상태 변경 Flow를 체험해보세요 (Notion 체험 가이드 참고)
3. **같은 개밥먹기 노션 문서**에 멤버 체험 내용도 이어서 기록
   - "워크스페이스 오너로는 이랬는데 멤버로는 이렇다" 식의 비교 메모가 특히 좋아요!

> 💡 완벽하게 쓸 필요 없어요! 키워드, 스크린샷, 한 줄 메모도 괜찮습니다.
> 🖥️ [Claude Code로 복귀] 워크스페이스 오너 + 멤버 체험이 **모두** 끝나면 "완료"라고 입력해주세요!

# Phase 2: AI 제품 개선안 (선택)

## EXPLAIN

### AI가 개밥먹기를 분석해줄 수 있어요

Phase 1에서 워크스페이스 오너/멤버 체험을 하면서 개밥먹기 노션 문서에 기록하셨죠?

여기서 한 가지 **보너스 체험**을 해볼 수 있어요:
Claude의 서브에이전트 3개(UX, 기술, 비즈니스)가 여러분의 개밥먹기 문서를 **동시에 분석**해서, 각 관점의 제품 개선안을 만들어주는 기능이에요.

이건 필수가 아니에요! 하지만 AI가 여러 관점에서 문서를 분석하는 과정을 체험해보고 싶다면 해볼 수 있어요.

완성되면 개밥먹기 노션 문서 안에 **AI 제품 개선안**이 하위 페이지로 생성됩니다:

```
📄 개밥먹기 — {닉네임}              ← 여러분이 작성 (Phase 1)
  └─ 📄 AI 제품 개선안            ← Claude가 자동 생성
```

> 💡 **원본은 그대로 보존돼요!** 여러분이 작성한 개밥먹기 원본 콘텐츠는 보존되며, AI 개선안은 {닉네임}의 개밥먹기 페이지 하단에 **별도 페이지**로 생성됩니다. 원본이 수정되거나 덮어씌워지지 않아요.

---

## EXECUTE

> 이번 Phase에서는 🖥️ **Claude Code 터미널**을 사용합니다.

> **에이전트 지시** (신규입사자에게 보여주지 않는 내부 메모):
>
> **이 Phase는 선택형이다.**
> 1. 개밥먹기 문서 작성이 완료되었는지 확인한다
> 2. AskUserQuestion으로 묻는다:
>    "개밥먹기 문서 작성은 잘 마무리하셨나요?
>    여기서 **보너스 체험**을 해볼 수 있어요 — AI 서브에이전트 3개(UX/기술/비즈니스)가
>    여러분의 개밥먹기 문서를 동시에 분석해서 제품 개선안을 만들어줍니다.
>    체험해보시겠어요? (해볼게요 / 다음 단계로 넘어갈게요)"
> 3. **"해볼게요"** → 3-A로 진행
> 4. **"넘어갈게요"** → Phase 2 스킵, progress.json 업데이트 후 Step 2 마무리로
>
> **3-A: 개밥먹기 문서 확인**
> 1. 신규입사자의 개밥먹기 노션 문서 페이지를 Notion에서 찾는다
>    - `notion-search`로 신규입사자 닉네임 + "개밥먹기" 키워드로 검색
>    - 또는 `onboarding_db`에서 해당 카드를 찾는다
> 2. `notion-fetch`로 문서 내용을 읽는다
> 3. 문서가 비어 있거나 내용이 너무 적으면:
>    - "체험하면서 느낀 점을 조금 더 적어주시면 AI 개선안의 품질이 올라가요!" 안내
>    - 작성을 기다린다 → STOP
> 4. 내용이 충분하면 바로 3-B로 진행한다
>
> **3-B: AI 개선안 생성 (병렬 분석)**
>
> **⚠️ 중복 감지 (생성 전 필수 확인)**
> 1. 개밥먹기 노션 문서의 하위 페이지에서 기존 AI 개선안이 있는지 확인한다:
>    - `notion-search`로 "AI 개선안" 키워드로 검색한다
>    - 검색 결과가 없으면 "AI 제품 개선안" 키워드로 한 번 더 검색한다
>    - 검색 결과 중 해당 개밥먹기 문서의 하위 페이지인 것만 필터링한다
> 2. **이미 존재하는 경우** → AskUserQuestion으로 묻는다:
>    "이전에 작성한 AI 개선안이 이미 있습니다. 어떻게 할까요?
>    (새로 작성할게요 → 기존 페이지 삭제 후 새로 생성 / 기존 것을 업데이트할게요 → 기존 페이지에 덮어쓰기)"
>    - **"새로 작성"** → 기존 페이지를 아카이브(`notion-update-page`로 archived: true)한 뒤 아래 1번부터 진행
>    - **"업데이트"** → 아래 1~2번으로 분석 결과를 만든 뒤, `notion-update-page`로 기존 페이지 내용을 덮어쓴다 (새 페이지 생성 안 함)
> 3. **존재하지 않는 경우** → 아래 1번부터 정상 진행
>
> 1. Agent 도구로 3개 서브에이전트를 **동시에** 실행한다:
>    - **UX/사용성 에이전트**: 사용자 경험, UI 흐름, 직관성, 접근성 관점 분석
>    - **기능/기술 에이전트**: 기능 완성도, 성능, 안정성, 기술적 개선 관점 분석
>    - **비즈니스 에이전트**: 워크스페이스 오너 수익화, 전환율, 리텐션, 경쟁력 관점 분석
>    - 각 에이전트에게 개밥먹기 문서 전문을 전달하고, 해당 관점의 개선 제안 3~5개를 요청한다
> 2. 3개 에이전트 결과를 통합한다:
>    - 관점 간 중복 제거 + 유사 제안 병합
>    - 영향도/난이도 기준 우선순위 정리
>    - 아래 "AI 개선안 구조"에 맞춰 최종 문서 작성
> 3. AI 개선안을 **개밥먹기 노션 문서 페이지의 하위 페이지**로 생성한다:
>    - `notion-create-pages`의 `parent` 파라미터에 개밥먹기 노션 문서 페이지 ID를 지정
>    - ❌ 온보딩 칸반과 별도로 생성하지 않는다 — 반드시 개밥먹기 노션 문서 아래에 넣는다
> 4. Notion 생성 실패 시 fallback: 로컬 `outputs/` 폴더에 저장하고 "나중에 Notion에 옮기면 됩니다" 안내
> 5. progress.json의 `step2_status`를 `"completed"`로 설정
> 6. 완료 후 신규입사자에게 안내한다:
>    - "{닉네임}이 작성하신 개밥먹기 문서 하단에 AI 개선안을 작성했어요!" 메시지
>    - 생성된 페이지의 Notion URL 공유
