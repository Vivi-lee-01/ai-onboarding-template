# 🤖 GitHub 사용 가이드 — AI 자동화 & 공용툴 레포지토리

## AI 산출물 Organization 주소: [github.com/teamspace-ai-all](http://github.com/teamspace-ai-all)

> 💡 **왜 GitHub을 사용하나요?**
> Claude Code 등 AI 도구로 만든 산출물(스크립트, 프롬프트, 설정 파일 등)을 한곳에 모아 관리하기 위해 회사 GitHub Organization을 운영합니다.
> - 내 컴퓨터에만 두면 수정 이력이 남지 않고, 이전 버전으로 되돌리기 어렵습니다
> - 공용 툴의 경우 다른 팀원이 활용하기 어렵습니다
> - 어떤 툴이 어떻게 만들어졌는지 확인할 수 있어야 합니다. :)

### 📖 GitHub 용어 정리

| 용어 | 뜻 |
|------|----|
| **Repository (레포)** | 파일들이 모여있는 프로젝트 폴더 |
| **Commit** | "이렇게 바꿨어요"라는 기록을 남기며 저장하는 것 |
| **Organization** | 회사 단위의 GitHub 공간 |
| **Clone (클론)** | GitHub 레포를 내 컴퓨터로 복사 |
| **Push (푸시)** | 내 변경사항을 GitHub에 올리기 |
| **Pull (풀)** | GitHub의 최신 내용을 내 컴퓨터로 가져오기 |
| **Branch (브랜치)** | 원본(main)을 건드리지 않고 독립적으로 작업할 수 있는 복사본 |
| **PR (Pull Request)** | 브랜치의 작업 내용을 main에 합치는 것을 요청. 협업 시 사용하며 해당 레포의 오너가 PR 확인 후 main에 반영합니다. |
| **.gitignore** | 민감 파일이 자동으로 올라가지 않게 막는 안전장치 |
| **Issue (이슈)** | 레포 안에서 버그 신고, 기능 요청, 할 일 등을 기록하는 게시판 같은 기능 |
| **Fork (포크)** | 다른 사람의 레포를 내 계정으로 통째로 복사. 원본에 영향 없이 자유롭게 수정 가능 |
| **Release (릴리스)** | 특정 시점의 코드를 "v1.0" 같은 버전으로 묶어서 공개하는 것 |

> 💡 프로덕트(Collabo) 개발용 GitHub과는 **완전히 별도의 공간**입니다.
> 📌 **모든 AI 산출물은 GitHub에 올려주세요.** 내 산출물은 바로 push하시고, 타인과 공통 작업물 또는 클론한 레포의 경우 PR하세요. Organization 안에서 자유롭게 레포를 생성하시면 됩니다(프로젝트별·목적별 권장).

### 시작하기

**Step 1: GitHub 가입**
1. [github.com](http://github.com) 접속 → Sign up → 이메일, 비밀번호, 사용자명 입력 → 이메일 인증
2. 완료 후 **회사 이메일(또는 GitHub username)을 Operations팀(Dana)에게 알려주세요.** 초대해 드립니다.
   - 이미 GitHub 계정이 있다면 바로 Step 2로.

**Step 2: 초대 수락**
1. 이메일 받은편지함에서 Organization 초대 메일 → Join 클릭
2. 끝! 레포지토리에 접근할 수 있습니다. (메일이 안 보이면 스팸함 또는 GitHub 🔔 알림 확인)

**Step 3: GitHub 보안 규칙 적용** — Claude Code를 열고 아래를 한 번에 복사 → 붙여넣기:
```bash
mkdir -p ~/.claude && cat >> ~/.claude/CLAUDE.md << 'TEAMSPACE_SECURITY'

# 팀스페이스 GitHub 보안 규칙
- .env, .env.* 파일은 절대 git에 커밋하지 마라
- API 키, 비밀번호, 토큰, 인증서 등 민감 정보를 코드나 파일에 포함하지 마라
- *.key, *.pem, credentials/, secrets/ 등 인증 관련 파일은 절대 커밋하지 마라
- 개인정보(이름, 연락처, 주민번호 등)가 포함된 파일은 절대 커밋하지 마라
- git push 전에 민감 정보가 포함되지 않았는지 반드시 확인하라
- 실수로 민감 파일을 커밋했다면 즉시 경고하고, 해당 키/비밀번호를 변경하도록 안내하라
- 커밋 메시지는 한글로 명확하게 작성하라 (예: "폰스크리닝 자동화 v2 추가")
- git push --force 는 절대 사용하지 마라
TEAMSPACE_SECURITY
```
> ✅ 최초 한 번만 실행하면 됩니다. 기존 설정에 영향 없이 보안 규칙이 뒤에 추가됩니다.

### 사용법 — Claude Code에게 말하면 끝

- **작업물을 GitHub에 올리고 싶을 때:** `지금까지 작업한 내용을 GitHub에 올려줘`
- **작업 시작 전 최신 내용 가져오기:** `{프로젝트명} 깃헙에서 최신 내용 가져와줘`
- **기존 레포를 처음 가져올 때(최초 1회):** `{레포이름} 레포를 클론해줘 (GitHub 레포 URL: https://github.com/teamspace-ai-all/{레포이름}.git)`

> 폴더 위치, 커밋 메시지 등은 Claude Code가 알아서 처리합니다.

### 협업 레포 팁: main 브랜치 보호

여러 명이 함께 쓰는 레포는 실수로 다른 사람의 작업을 덮어쓸 수 있습니다. 레포 루트에 `CLAUDE.md`를 만들고 아래 규칙을 추가하면 Claude Code가 자동으로 PR 방식으로 작업합니다:
```
# 협업 규칙
- main 브랜치에 직접 push하지 마라. 반드시 별도 브랜치에서 작업 후 PR(Pull Request)로 머지하라
```
추가로 Git hook으로 물리적 차단도 가능: `main 브랜치에 직접 push 못하도록 pre-push hook 설치해줘`
> 💡 혼자 쓰는 레포에는 main 브랜치 보호가 꼭 필요하진 않습니다. 협업 레포에만 추가하세요.

## ⚠️ 이것만 주의하세요

**절대 올리면 안 되는 것:** API 키, 비밀번호, `.env` 파일, 개인정보가 포함된 파일
→ 한번 올라가면 삭제해도 이력에 남습니다. 실수로 올렸다면 즉시 Operations팀(Dana)에게 연락하세요.
> Step 3에서 설정한 CLAUDE.md 규칙이 자동 적용되어 대부분 방지됩니다. 레포 생성 시 `.gitignore`도 함께 추가 권장.

## 문의
GitHub 관련 문의(가입/초대, 권한 요청, 파일 실수 복구 등)는 **Operations팀 Dana**에게 연락해주세요.

## 레포지토리(Repository) vs 프로젝트(Project)

| | **Repository (레포)** | **Project (프로젝트)** |
|---|---|---|
| **역할** | 코드와 파일을 저장하는 공간 | 작업(이슈, PR)을 관리하는 칸반 보드 |
| **비유** | 서류 캐비닛 (파일 보관) | 화이트보드 (진행 상황 관리) |
| **포함 내용** | 코드, 스크립트, 설정 파일, 변경 이력 | 할 일, 진행 중, 완료 등 작업 상태 |
| **우리가 주로 쓰는 것** | ✅ 레포 — AI 산출물 저장 | 필요 시 활용 (노션과 역할 유사) |

**요약:** 레포 = 파일 저장소, 프로젝트 = 일감 관리 도구. 우리는 주로 레포를 사용하고, 일감 관리는 노션에서 합니다.
